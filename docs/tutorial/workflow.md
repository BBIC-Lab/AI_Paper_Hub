# 推荐链路说明

推荐链路负责把当天抓取到的论文变成首页和 Daily Papers 中的精读、速读内容。理解这条链路后，你可以更快判断问题出在论文源、查询召回、排序融合、LLM 过滤，还是最终文档生成。

【流程图占位：论文源 -> 召回 -> RRF 融合 -> LLM refine -> 精读/速读选择 -> Docsify 页面】

## 一句话流程

系统先从多个论文源抓取候选，再用订阅查询做 BM25 和向量召回，接着用 RRF 融合排序，并交给 LLM 做相关性与摘要判断，最后按精读区和速读区规则生成网页。

## Step 1：论文源抓取

论文源阶段负责把 arXiv、会议源或其他已启用来源的新论文同步到候选池。每篇论文需要保留标题、作者、摘要、来源、链接、发布日期等元数据。

你需要关注：

- 今天是否成功抓到候选论文。
- 论文的来源字段是否完整。
- PDF 链接或外部链接是否可用。

【截图占位：抓取阶段日志与候选数量统计】

## Step 2：订阅查询召回

召回阶段会读取你在订阅中写的主题描述，并从候选池里找出可能相关的论文。通常会同时使用关键词匹配和语义相似度，避免只靠单个关键词造成漏选。

如果相关论文没有进入候选，优先检查查询描述是否太窄、关键词是否缺少同义表达、论文源是否覆盖该方向。

【截图占位：某个订阅命中的候选论文列表】

## Step 3：RRF 融合排序

BM25 和向量召回会产生不同排序。RRF 会把多路排序融合成一个更稳的候选列表，让既有关键词命中又有语义相关性的论文更靠前。

排查建议：

- 如果标题关键词很准但语义不相关，检查排除方向。
- 如果语义相关但关键词缺失，补充任务名、数据集名或应用场景。
- 如果多个订阅都命中同一篇论文，说明它可能是跨主题结果，需要在日报中重点阅读推荐理由。

【截图占位：BM25、Embedding 与 RRF 排名对比表】

## Step 4：LLM 相关性判断与摘要

LLM refine 阶段会进一步判断论文和订阅之间的关系，并生成更适合阅读的推荐理由、摘要和标签。这里最依赖模型质量，也最容易受到配置成本影响。

建议：

- 初次配置先用一个稳定模型跑通。
- 当召回质量稳定后，再把总结或精读任务切换到更强模型。
- 如果推荐理由经常空泛，优先优化订阅描述，而不是立刻更换模型。

【截图占位：LLM refine 输出中的推荐理由与标签】

## Step 5：精读区与速读区选择

选择阶段会先保留精读区论文，再用剩余额度填充速读区。这样可以确保最重要的论文不会被数量上限挤掉，同时保留一部分可能有用的扩展阅读。

你可以通过两个上限控制日报长度：

- 精读区上限：影响最核心的深读内容。
- 速读区上限：影响扩展阅读和备选论文数量。

【截图占位：日报中精读区与速读区分区展示】

## Step 6：生成 Docsify 网页

最后，系统会把结果写入 docs 目录，首页展示最新日报，Daily Papers 保存按日期组织的历史记录，使用教程则作为固定帮助入口保留。

如果 GitHub Pages 已开启，合并到 main 后网页会随仓库内容更新。

## 快速排查表

| 问题 | 优先检查阶段 | 处理方式 |
| --- | --- | --- |
| 今天没有新论文 | 论文源抓取 | 看 Actions 日志和候选数量 |
| 有论文但不相关 | 订阅召回 | 收紧查询描述，增加排除方向 |
| 相关论文排序靠后 | RRF 融合 | 补充关键词或语义描述，提高多路命中机会 |
| 推荐理由不好 | LLM refine | 优化订阅描述，必要时更换模型 |
| 页面没有更新 | Docsify 生成与 Pages | 确认 docs 文件已提交，Pages 部署完成 |

如果你已经能定位问题阶段，就回到 [订阅与查询配置](/tutorial/configuration) 调整对应字段，再手动运行一次日报验证。
